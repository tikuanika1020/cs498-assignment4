Q1: Decoupling TMS and Storage
FoundationDB decouples these layers to allow them to scale independently based on the workload’s needs; for instance, you can scale the Transaction Management System (TMS) to handle high commit rates without necessarily increasing disk capacity. The TMS is a stateless, distributed system responsible for transaction ordering and conflict detection, while the Storage System (SS) is responsible for persistently storing data and serving read requests from local disks.
Q2: Conflict Detection
The Resolver component within the TMS is responsible for detecting read-write conflicts by comparing the read versions of a committing transaction against the commit versions of recently finished transactions. This check occurs during the optimistic concurrency control phase, specifically after the transaction has been submitted for commit but before it is finalized and written to the transaction logs.
Q3: Read Versions and Consistency
The read version acts as a logical timestamp that ensures a transaction sees a globally consistent "snapshot" of the database at a specific point in time. By providing this version, the Sequencer guarantees Strict Serializability, as it ensures that the transaction will not see partial results from other concurrent commits or any data written after its assigned read version.
Q4: Snapshot Reads vs. Standard Reads
Standard reads are tracked by the Resolver and will cause a transaction to fail if the data is modified by another transaction before the commit. In contrast, snapshot reads bypass conflict detection entirely, allowing the transaction to succeed even if the underlying data changed. You would choose snapshot reads for operations where absolute consistency is less critical than avoiding "hot spot" conflicts, such as generating an approximate aggregate count or performing background telemetry.
Q5: "Cheap" Recovery
FDB makes recovery cheap by ensuring that the LogServers have already persisted all acknowledged mutations in a way that is easily accessible to the new generation of Proxies and Resolvers. Because the StorageServers pull data from the LogServers asynchronously, the system only needs to identify the last stable "end of the log" to resume operations, rather than scanning and re-applying a massive redo log to the entire disk state during the critical path of startup.
